Local vs Remote

and working between the two

So you've added your code to your machine and you are making changes. There is an entire process to getting your code save and uploaded back to GitHub.

Here are a few things that you will need to know before we continue.

  • The repository that you got your code from is the origin.
  • Any repository not on your computer) is referred to as a remote repository.
  • Your local machine's code is simply referred to as the local repository.

By default your code will always work with the local and origin repositories.

Fetching and Pulling

When you are working with multiple people on a project, I highly recommend that you check the origin for any changes. This will prevent you from running into any major CONFLICTS later on.

To check for any changes you will need to run a fetch on origin.

git fetch

If pulling changes from a branch other than origin/and the same branch you are on you, you will need to include it.

git fetch remote <some-users-repo/their-branch> To update your code with the origin you will need to git pull information down from your origin (or some other) repository. A pull, checks the code for any changes and then immediately tries to merge those changes with your local copy.

Commit

When you are making changes, part of version control is making the versions. These are known as commits. Commits are records or reference that you can point to in your code.

It is recommended that you commit often and make your commits logical.
